# CBCTracker

A longitudinal Complete Blood Count (CBC) trend analyzer built for oncology clinical decision support. The system uses a **Temporal Convolutional Network (TCN)** to detect anomalous patterns in sequential CBC blood draw data, helping oncologists identify early signs of treatment failure, relapse, or acute complications.

---
## Inspiration

I have worked as a medical assistant in an Oncology office and have had the experience of understanding the daily workflow of an oncologist firsthand. We often order Complete Blood Count (CBC) tests for patients which are critical in monitoring treatment side effects and the potential spread of cancer. While Oncologists are highly trained in reading the values (ex. WBC, ANC, RBC, HGB, PLT, LYM, MONO) in this tests, it is easy to become overwhelmed due to needing to monitor hundreds of patients with multiple CBC tests, especially as patients have different baseline values and risk levels. This tool was created to help oncologists organize and track the CBC values for patients diagnosed with cancer over a comprehensive time period. 

CBCTracker uses accumulated CBC patient data to assign anomaly risks levels based on patient baseline values. By analyzing the severity of anomaly risk in each patient, Oncologists can make informed decisions on whether or not to switch or keep current cancer treatment plans.

## Overview

Cancer patients undergoing chemotherapy require frequent CBC monitoring. Manual review of longitudinal lab trends across dozens of patients is time-consuming and prone to missing subtle early warning signals. This tool automates that surveillance by:

- Learning each patient's **personal CBC baseline** and expected nadir/recovery pattern
- Flagging draws that deviate significantly from that individual trajectory
- Classifying the **type of anomaly** detected (sudden drop, sustained elevation, erratic pattern)
- Providing a clinical dashboard for managing patients and reviewing trends

> **Important:** This project is a research prototype trained on synthetic data. It is not validated for clinical use. All predictions should be reviewed by a qualified clinician.

## Anomaly Types Detected

| Pattern | Clinical Significance | Example Cause |
|---|---|---|
| **Sudden drop** | Acute severe suppression | Aplastic crisis, febrile neutropenia, sepsis |
| **Sustained elevation** | Persistent counts above normal | Infection, disease relapse, steroid response |
| **Erratic** | High-variance instability | Treatment failure, marrow engraftment failure |
| **None** | Expected chemo nadir/recovery | Normal treatment response |

## Model Architecture

The model is a **multi-input TCN** with three components:

```
x_seq    (batch, T=15, 8 features)   ← CBC values + log time delta
    └─ permute → TCNBackbone (4 dilated causal conv blocks)
                 dilations: [1, 2, 4, 8]  receptive field: 31 timesteps
                 → (batch, 64, T)

x_static (batch, 4 features)          ← age, sex, cancer type, chemo protocol
    └─ StaticFeatureBranch MLP
       → (batch, 64, 1)  [broadcast-added to TCN output]

Fused → AnomalyHead (pointwise conv 64→32→1)
      → sigmoid → anomaly probability per timestep (batch, T)
```

**Key design decisions:**

- **Causal convolutions** — each output timestep only sees past draws, never future ones. The model can score a draw the day it arrives without needing subsequent data.
- **Dilated convolutions** — dilation doubles at each block (1, 2, 4, 8), giving exponentially growing receptive field without extra parameters.
- **Patient-level train/val/test split** — prevents data leakage between splits.
- **Weighted BCE loss** — `pos_weight ≈ 2.3` compensates for class imbalance (~30% anomalous draws).

## Quickstart

### 1. Install dependencies

```bash
python -m venv .venv
source .venv/bin/activate        # Windows: .venv\Scripts\activate
pip install -r requirements.txt
```

### 2. Generate synthetic training data

```bash
python phase1_generate_data.py
```

Generates `data/cbc_synthetic.csv` — 500 patients, ~20,000 CBC draws, four anomaly types.

### 3. Train the model

```bash
python train.py
```

Runs preprocessing, trains for up to 50 epochs with early stopping, saves `outputs/checkpoints/best_model.pt`. Typical training time: 3–8 minutes on CPU, under 1 minute on Apple Silicon (MPS).

Expected results on synthetic data:

| Metric | Value |
|---|---|
| AUROC | ~0.96 |
| AUPRC | ~0.92 |
| Sensitivity | ~0.78 |
| Specificity | ~0.97 |

### 4. Evaluate

```bash
python phase5_part1_evaluate.py
```

Runs the trained model on the held-out test set and saves `outputs/evaluation_results.json`.

### 5. Launch the clinical dashboard

```bash
python seed_demo_patients.py    # load 16 demo patients (optional)
streamlit run app.py
```

## Clinical Dashboard

The Streamlit app has two tabs:

**Patient Management**
- Create new patients with demographic info and first CBC draw
- Update existing patients with new draws over time
- Delete individual draws or remove a patient entirely
- Load 16 pre-built demo patients covering all four anomaly types

**Analytics**
- Select any patient and run anomaly scoring
- Interactive dual-axis chart: CBC feature overlay vs anomaly score
- Anomaly type badge (sudden drop / sustained elevation / erratic / none)
- Alert banner listing high-risk draw days
- All 7 CBC features mini overview grid
- Download full patient report as CSV

## Inference on New Patients

```bash
python predict.py --patient_csv path/to/patient.csv
```

The CSV must contain columns: `patient_id`, `timestamp_day`, `time_delta`, `WBC`, `ANC`, `RBC`, `HGB`, `PLT`, `LYM`, `MONO`, `age`, `sex`, `cancer_type`, `chemo_protocol`.

Minimum 15 draws required for scoring (window size). Output includes per-draw anomaly scores and risk levels (low / medium / high).

## Limitations

**Synthetic data.** The model was trained on procedurally generated data grounded in published clinical reference ranges and chemotherapy pharmacology, not on real patient records. Real CBC trajectories have wider inter-patient variability, disease-specific baseline effects, and comorbidity confounders that the synthetic data does not capture.

**Cancer-agnostic.** The model applies the same anomaly thresholds regardless of cancer type or chemotherapy protocol. In practice, AML patients have very different expected CBC ranges from breast cancer patients, and R-CHOP produces a different nadir profile than FOLFOX. Retraining on real data (e.g. MIMIC-IV) would allow the model to learn protocol-specific behaviour.

**No relapse detection.** The model detects statistical deviation from a patient's own CBC trajectory. It does not directly predict relapse, treatment failure, or clinical outcomes — those require labelled outcome data not present in the current training set.

**Threshold sensitivity.** The default anomaly threshold is 0.5. Clinical deployment would require calibration to a specific sensitivity/specificity operating point (e.g. sensitivity ≥ 0.90) validated against real outcome data.

## Extending to Real Data

The pipeline is designed to accept real data with minimal changes. The recommended path:

1. **MIMIC-IV** (free after credentialing at physionet.org) — rich longitudinal CBC data for hospitalised patients including cancer cohorts. Requires CITI training (~2 hours) and a data use agreement.

2. **TCGA clinical data** — open-access outcome labels (vital status, treatment response) available immediately at portal.gdc.cancer.gov. CBC lab values require dbGaP controlled-access application.

3. **De-identified EMR export** — if working with an institutional oncology practice, de-identified CBC exports slot directly into `phase2_preprocessing.py` with no architecture changes.

The key additional requirement for real data is **outcome labelling** — linking CBC trajectories to clinical events (hospitalisation, protocol changes, confirmed relapse) to enable supervised outcome prediction beyond anomaly detection.
